# План реализации: апгрейд whisper_own (macOS PTT Whisper dictation)

## 1. Что строим и главный принцип

Мы превращаем рабочий, но «ручной» инструмент диктовки в устанавливаемый-одной-командой, самонастраивающийся под железо, отказоустойчивый продукт с историей, меню-баром и умной очисткой текста — **12 фич**, сгруппированных по зависимостям.

**Guiding principle (незыблемо):**
- **Только macOS.** На других ОС — graceful exit, без установки.
- **Кольцевой буфер священен.** Математику смещений в `startCapture`/`finishCapture` (init.lua:463–529) и остановку рекордера строго через `interrupt()`/SIGINT (не SIGKILL) НИКТО не трогает. Все 12 фич аддитивны к этому ядру.
- **Groq-first остаётся быстрым путём** на слабом Mac; локальный whisper.cpp — всегда доступный офлайн-fallback.
- **IPC-контракт неизменен:** `/tmp/dictation.status` (`running|done|ignored|error:<reason>`), `/tmp/dictation.txt`, `/tmp/dictation.err`, `/tmp/dictation-whisper.pid`. init.lua поллит только status, stdout не парсит — поэтому shell-сторона может добавлять новые файлы свободно.
- **RU+EN mixed** — дефолт распознавания. Русский кириллицей, английский латиницей, без перевода.
- **Каждая фича заканчивается `/verify`** — реальной диктовкой через fn, а не только тестами.

## 2. Milestone 0 — «Общие основы» (shared foundations)

Несколько фич зависят от общих артефактов. Здесь мы фиксируем ОДИН согласованный дизайн и разрешаем расхождения между спеками. **Эти решения — источник истины; каждый loopTaskPrompt ниже уже приведён к ним.**

### 2.1. Единый файл детектора: `dictation-detect.sh`

Расхождение спеков: installer-wizard упоминает `hardware-detect.sh`, hardware-autodetect создаёт `dictation-detect.sh`.
**Решение:** канонический файл — **`/Users/yauhenishauchenka/whisper_own/dictation-detect.sh`**. Установщик вызывает именно `dictation-detect.sh --write-profile`. Имя `hardware-detect.sh` НЕ используется нигде.

### 2.2. `profile.env` — единственный носитель ФАКТОВ о железе

Путь: **`~/.local/share/whisper/profile.env`**. Пишется `dictation-detect.sh --write-profile`, сорсится в `dictation-transcribe.sh` ПЕРЕД блоком конфига. Идиома `${VAR:-...}` во всех строках → приоритет **runtime env > profile.env > policy > встроенные дефолты**, повторный source идемпотентен.

Расхождение: hardware-autodetect эмитит `DICTATION_ENGINE_ORDER/WHISPER_THREADS/...`, а model-policy ждёт факты `DICT_TIER/DICT_HAS_MLX/DICT_HAS_METAL_WHISPER`; installer ждёт `CHIP/RAM_GB/REC_MODEL_URL/REC_MODEL_PATH/REC_BUILD_LOCAL`.
**Решение — profile.env несёт ОБЪЕДИНЕНИЕ полей (факты + удобные производные + installer-поля):**

```sh
# Auto-generated by dictation-detect.sh on <date>
# Precedence: runtime env > this profile > policy > dictation-transcribe.sh defaults

# --- facts (read by model-policy.resolve_model_policy) ---
export DICT_TIER="${DICT_TIER:-weak}"              # apple-strong | apple-capable | weak
export DICT_HAS_MLX="${DICT_HAS_MLX:-0}"           # mlx_whisper на PATH
export DICT_HAS_METAL_WHISPER="${DICT_HAS_METAL_WHISPER:-0}"

# --- human/installer facts ---
export DICT_CHIP="${DICT_CHIP:-unknown}"
export DICT_ARCH="${DICT_ARCH:-arm64}"
export DICT_RAM_GB="${DICT_RAM_GB:-0}"
export REC_MODEL_URL="${REC_MODEL_URL:-https://huggingface.co/ggml-org/whisper.cpp/resolve/main/ggml-large-v3-turbo-q5_0.bin}"
export REC_MODEL_PATH="${REC_MODEL_PATH:-$HOME/.local/share/whisper/ggml-large-v3-turbo-q5_0.bin}"
export REC_BUILD_LOCAL="${REC_BUILD_LOCAL:-1}"     # 0 на слабом Mac -> Groq-only

# --- pre-resolved recommendations (policy re-derives from facts if unset) ---
export DICTATION_ENGINE_ORDER="${DICTATION_ENGINE_ORDER:-groq whisper.cpp}"
export WHISPER_THREADS="${WHISPER_THREADS:-4}"
export DICTATION_COMPUTE="${DICTATION_COMPUTE:-cpu}"
export DICTATION_LLM_CLEANUP="${DICTATION_LLM_CLEANUP:-0}"

# MODEL_PATH — только если файл реально существует
```

**Важно про ffmpeg-пути:** `FFMPEG_PATH`/`FFPROBE_PATH` **не** эмитятся `dictation-detect.sh` — их ДОПИСЫВАЕТ установщик через `brew --prefix` (на Intel это `/usr/local`, на Apple Silicon `/opt/homebrew`). Детектор НЕ должен хардкодить `/opt/homebrew` в профиль слабого Intel-Mac. Строки FFMPEG появляются в profile.env только после запуска `install.sh`.

`dictation-detect.sh` эмитит и факты `DICT_*`, и pre-resolved значения по ОДНОЙ детерминированной таблице (см. 2.4). `resolve_model_policy()` использует ту же таблицу для гэп-филла в рантайме, поэтому они согласованы по построению. Каноническая модель (`ggml-large-v3-turbo-q5_0.bin`) и её URL совпадают с дефолтом `MODEL_PATH` в transcribe.sh:36.

### 2.3. Контракт engine-dispatcher

- `DICTATION_ENGINE_ORDER` — space-separated, токены **`mlx | whisper.cpp | groq`** (алиасы: `mlx-whisper`, `local`, `cloud`). Дефолт `groq whisper.cpp` = сегодняшнее поведение.
- **Канонические имена движков для `$ENGINE`, `/tmp/dictation.engine`, истории, меню-бара и доков — это токены `mlx | whisper.cpp | groq`** (НЕ старое значение `local`). Победивший токен из `DICTATION_ENGINE_ORDER` пишется в `$ENGINE` дословно.
- Каждый движок — унифицированный контракт: пишет финальный текст в `OUT_PATH`, возвращает **0** = успех, **2** = движок недоступен → пропустить молча, **иное != 0** = ошибка → fallback к следующему.
- Победивший движок пишется в **`/tmp/dictation.engine`** (`ENGINE_PATH`). Это единственный новый IPC-файл; init.lua его не читает — читают diagnostics, doctor и docs.

### 2.4. Каноническая таблица политики движка/модели (одна на всех)

| Tier | Условие | DICTATION_ENGINE_ORDER | Локальная модель (pref) |
|---|---|---|---|
| **apple-strong** + mlx | arm64, RAM≥16, mlx present | `mlx whisper.cpp groq` | turbo → turbo-q5 → large-v3 |
| **apple-strong / apple-capable** | arm64, whisper.cpp собран | `whisper.cpp groq` | turbo-q5 → turbo → large-v3 |
| **weak** | Intel / мало RAM / нет Metal-сборки | `groq whisper.cpp` | turbo-q5 (fallback) |
| пусто (нет ключа, нет сборки) | — | `groq whisper.cpp` | turbo-q5 (default path) |

- Универсальная модель = `ggml-large-v3-turbo-q5_0.bin` (латентность важнее; RU turbo≈large-v3). `large-v3` только опционально при strong + RAM≥24.
- Облако: `GROQ_MODEL=whisper-large-v3`.
- `resolve_model_policy()` выставляет `DICTATION_ENGINE_ORDER/MODEL_PATH/GROQ_MODEL` **только если они пустые** (runtime env всегда побеждает).

### 2.5. История: `~/.local/share/whisper/history.jsonl`

Пишет ТОЛЬКО shell-воркер (в ветке успеха, через `$ENGINE` in-process — файл `/tmp/dictation.engine` для истории НЕ обязателен, но существует для диагностики). Формат — одна JSON-строка на диктовку, старые сверху, cap = `DICTATION_HISTORY_MAX` (дефолт 50; `<=0` отключает), `chmod 600`, атомарная запись `os.replace`, `ensure_ascii=False`.

Поле `engine` содержит канонический токен движка (`groq | whisper.cpp | mlx`), а НЕ старое `local` — см. 2.3. Пример с локальным движком:

```json
{"ts":1752406325,"iso":"2026-07-13T14:32:05+03:00","text":"Проверка раз.","engine":"whisper.cpp","mode":"cut","dur":2.14}
```

**Читающий контракт (Lua):** глобальная функция **`dictationHistoryRead(limit)`** (дефолт 20), newest-first, `pcall(hs.json.decode)` построчно, битые строки пропускает.
Расхождение имён устранено: menubar вызывает именно **`dictationHistoryRead`** (не `dictationHistoryEntries`); «Clear history» делает `os.remove(historyPath)` (отдельная `dictationHistoryClear` не вводится).

### 2.6. init.lua: конфиг, состояние, точки-хуки

- **Новые config-локалы** (верхний блок, строки 20–23): `triggerMode="ptt"`, `toggleMaxSeconds=300`, `toggleStartAlert=true`, `hotkeyWatchdogInterval=2`, `menubarEnabled=true`, `menubarHistoryCount=10`; палитра `menubarColors` (ОТДЕЛЬНАЯ от `indicatorColors`); `historyPath`.
- **`triggerMode` — единственный локал** (не глобал `dictationTriggerMode`). Меню-бар читает `triggerMode` напрямую (в одном файле это upvalue; при отсутствии — global-nil-lookup → дефолт «Push-to-talk»). Устраняет расхождение toggle↔menubar.
- **5 indicator-choke-points** (211–225, 181–189) — единственный источник состояния и для экранной точки, и для иконки меню-бара: `indicatorShowRecording→recording`, `indicatorShowProcessing→transcribing`, `indicatorShowSuccess/Error/Hide→idle`.
- **Глобальные объекты, переживающие reload:** `dictationFnTap`, `dictationRecorderWatchdog`, `dictationWakeWatcher` (существуют); добавляются `dictationMenubar`, `dictationHotkeyWatchdog`.

### 2.7. Порядок пост-обработки в ветке успеха shell (финальный, согласованный)

```
engine dispatch → RAW_OUTPUT capture → write ENGINE_PATH ($ENGINE → /tmp/dictation.engine) →
clean_hallucinations (static) → STATIC_OUTPUT capture →
llm_postprocess (flag-gated, fail-open) →
log_diagnostics(engine, raw, static) → append_history → printf 'done' > STATUS_PATH
```

**Инвариант всех переписываний ветки успеха:** строка `printf '%s' "$ENGINE" > "$ENGINE_PATH"` ставится СРАЗУ после выбора движка (до `clean_hallucinations`) и НЕ должна теряться при последующих правках блока (B1, C1 переписывают тот же блок — обязаны её сохранять).

`append_history` — ПОСЛЕДНИЙ шаг с текстом (пишет уже LLM-очищенный финал). Каждый новый шаг fail-open: статус `error` только если УПАЛИ все движки.

## 4. Рекомендуемый порядок `/loop`

```
Шаг 1  ┌─ A1 hardware-autodetect ─┐   ┐
       │  A2 engine-dispatcher    │   │ параллельно (нет общих зависимостей;
       │  B1 llm-postprocess      │   │ B1/C3/C4 трогают разные части)
       │  C3 toggle-mode          │   │
       └─ C4 hotkey-watchdog ─────┘   ┘   (все правят разные файлы/секции)

Шаг 2   A3 model-policy         (после A1 + A2)
        A4 language-default     (после A2; B1 желателен раньше для LLM-критерия)

Шаг 3   C1 dictation-history    (после A2 — нужен $ENGINE)

Шаг 4   C2 menubar-icon         (после C1)

Шаг 5   D1 installer-wizard     (после A1)

Шаг 6   D2 skill-packaging      (после D1 + D3)   ← D3 должен быть готов
        D3 docs                 (после ВСЕХ реализующих: A1-A4, B1, C1-C4, D1)
```

**Практический последовательный прогон (безопасный для одного исполнителя):**
1. A1 → 2. A2 → 3. A3 → 4. **B1** → 5. **A4** → 6. C1 → 7. C3 → 8. C4 → 9. C2 → 10. D1 → 11. D3 → 12. D2.

> Порядок B1 перед A4 гарантирует, что условный критерий A4 «LLM сохраняет оба алфавита» проверяем на месте (A4 формально зависит только от A2, но выигрывает от уже приземлившегося B1).

**Что реально можно распараллелить** (разные файлы, нет общих правок): {A2, B1, C3, C4} на ранней стадии; A1 отдельно (создаёт detect + одна строка в transcribe.sh). Осторожно с одновременными правками `dictation-transcribe.sh` (A2/A3/A4/B1/C1 все его трогают) — их лучше сериализовать в порядке config-блока, чтобы избежать конфликтов слияния. **Особый инвариант при сериализации:** строка `printf '%s' "$ENGINE" > "$ENGINE_PATH"` (введена A2) обязана уцелеть при переписывании ветки успеха в B1 и C1 — см. 2.7. init.lua-фичи (C2/C3/C4) независимы по секциям и мержатся чисто.
